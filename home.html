01 — Fundamentals

DBMS vs DBaaS

Two paradigms for storing and managing data — one gives you control, the other gives you convenience.

DBMS — Self-Managed

What it is: Software installed on your own servers (MySQL, PostgreSQL, Oracle). You own the full stack.

Infrastructure: You provision hardware, configure OS, tune performance, and manage storage.

Control: Full control over query optimization, storage engines, indexing strategies, and network topology.

Cost model: High CapEx upfront. Lower cost at large scale with amortized hardware.

Best for: On-premise enterprises, regulated industries, latency-sensitive workloads, and custom configurations.

DBaaS — Cloud-Managed

What it is: Database hosted and maintained by a cloud provider (Amazon RDS, Azure SQL, Google Cloud SQL).

Infrastructure: Provider handles hardware, OS patches, failover, backups, and scaling automatically.

Control: Abstracted infrastructure. You focus on data and queries, not servers.

Cost model: OpEx pay-as-you-go. Scales costs with usage — efficient for variable workloads.

Best for: Startups, SaaS products, teams that need to ship fast without dedicated database engineers.

-- Self-managed: you write the config
CREATE DATABASE myapp WITH ENCODING 'UTF8';
ALTER SYSTEM SET max_connections = 500;
-- DBaaS: the cloud just handles it
$ aws rds create-db-instance \
    --db-instance-identifier myapp-db \
    --db-instance-class db.t3.medium \
    --engine postgres
02 — Timeline

Database Evolution

From paper ledgers to globally distributed cloud databases — six decades of innovation in data management.

1950s–1960s
Pre-Database Era — Manual Systems
Data was stored in physical filing cabinets or magnetic tape. Retrieval was labor-intensive, error-prone, and not machine-optimized. No concurrent access, no structured querying. Programs and data were tightly coupled.
1960s–1970s
Navigational Era — Hierarchical & Network Models
IBM introduced IMS (Information Management System) using a hierarchical model. CODASYL defined the network model with DBTG. Data had to be traversed via explicit pointers — fast for known access patterns, rigid for everything else. No ad hoc queries.
1970s–1990s
Relational Era — SQL Dominates
Edgar Codd's 1970 paper "A Relational Model of Data for Large Shared Data Banks" changed everything. Tables, keys, and joins replaced pointer-based navigation. IBM's System R and Oracle's RDBS brought SQL to market. ACID guarantees made relational databases the backbone of enterprise computing.
2000s–2010s
NoSQL Era — Web Scale
Amazon's Dynamo paper (2007) and Google's Bigtable sparked a revolution. Facebook, Twitter, and LinkedIn needed horizontal scalability that RDBMS couldn't provide cheaply. MongoDB, Cassandra, Redis, and CouchDB emerged — sacrificing ACID for availability and partition tolerance (CAP theorem).
2012–Present
NewSQL & Cloud Native
Google Spanner (2012) proved you could have globally distributed, strongly consistent, SQL-compatible databases. CockroachDB and TiDB followed. DBaaS exploded. Serverless databases like PlanetScale and Neon emerged. AI-assisted query optimization and vector databases for LLM workloads are the latest frontier.
03 — Paradigms

The Core Trio

Three fundamental approaches to structuring and querying data — each a product of different engineering tradeoffs.

SQL / Relational
Relational
Data organized into tables with rows and columns. Relationships enforced via foreign keys. Queries written in declarative SQL. The gold standard for transactional systems requiring strict data integrity.
consistencyACID
scalingVertical
schemaRigid / Fixed
query langSQL
use casesFinance, ERP, CMS
examplesPostgreSQL, MySQL, Oracle
NoSQL
Non-Relational
Umbrella term for databases that abandon the table model: document (JSON), key-value, wide-column, and graph. Designed for horizontal scaling, flexible schemas, and high-throughput workloads.
consistencyBASE / Eventual
scalingHorizontal
schemaFlexible / Dynamic
query langVaries per DB
use casesBig Data, Real-time, IoT
examplesMongoDB, Redis, Cassandra
NewSQL
Distributed SQL
Best-of-both-worlds systems providing the consistency and familiar SQL interface of relational databases with the horizontal scalability of NoSQL. Built for geo-distributed, cloud-native deployments.
consistencyACID + Distributed
scalingHorizontal
schemaStructured / SQL
query langSQL-Compatible
use casesGlobal SaaS, Fintech
examplesSpanner, CockroachDB, TiDB
04 — Platforms

Top Database Platforms

The most widely deployed and technically capable database systems in production today.

SQL · Open Source
PostgreSQL
The world's most advanced open-source relational database. Supports advanced data types including JSON/JSONB, full-text search, geospatial via PostGIS, custom extensions, and procedural languages. Multi-version concurrency control (MVCC) ensures high read throughput without locking.
MVCC JSONB PostGIS PL/pgSQL Extensions WAL
SQL · Enterprise
Oracle Database
The dominant enterprise database for mission-critical workloads. Features Real Application Clusters (RAC) for active-active failover, Autonomous Database with ML-driven self-tuning, and decades of battle-tested reliability in banking, telecom, and government sectors.
RAC Autonomous DB Exadata In-Memory Multitenant
SQL · Web-Scale
MySQL
The most widely deployed open-source RDBMS on the web. Powers Wikipedia, YouTube, and countless WordPress sites. InnoDB storage engine provides ACID compliance. MySQL 8.0 introduced window functions, CTEs, and improved JSON support. Now maintained by Oracle with a vibrant community fork in MariaDB.
InnoDB Replication Group Replication JSON Window Functions
NoSQL · Document
MongoDB Atlas
A fully managed cloud document database platform built on MongoDB's BSON document model. Supports multi-cloud deployments across AWS, GCP, and Azure. Features Atlas Search (full-text), Atlas Vector Search for AI embeddings, Charts, and Data API for serverless access.
BSON Atlas Search Vector Search Aggregation Change Streams
Cloud · Data Warehouse
Snowflake
A purpose-built cloud data warehouse with a unique multi-cluster shared data architecture that separates storage from compute. Virtual warehouses can be independently scaled without downtime. Time Travel and Fail-safe features provide data recovery. The Data Cloud model enables secure cross-organization data sharing.
Multi-cluster Time Travel Data Sharing Snowpark Zero-Copy Cloning
Key-Value · In-Memory
Redis
Sub-millisecond in-memory data store used as a cache, message broker, and session store. Supports rich data structures: strings, hashes, lists, sets, sorted sets, streams, and geospatial indexes. Redis Sentinel provides high availability; Redis Cluster enables sharded horizontal scaling.
In-Memory Pub/Sub Streams Lua Scripting Cluster Mode

06 — Open Problems

Modern Challenges

Database engineers in 2025 face a new generation of technical and organizational problems.

01
Data Volume & Velocity
IoT devices, event streams, and user telemetry generate petabytes of data daily. Traditional batch ETL pipelines can't keep up. Solutions like Apache Kafka, Flink, and streaming databases (RisingWave, Materialize) are redefining real-time data pipelines. Tiered storage (hot/warm/cold) is essential for cost management at scale.
02
Security, Privacy & Compliance
GDPR, HIPAA, SOC 2, and PCI-DSS impose strict data residency, encryption, and audit requirements. Column-level encryption, row-level security, dynamic data masking, and transparent data encryption (TDE) are now table stakes. Zero-trust database access and secrets management via Vault are growing requirements for regulated workloads.
03
Multi-Cloud & Hybrid Complexity
Enterprises increasingly run databases across AWS, Azure, GCP, and on-premise simultaneously. Managing schema migrations, cross-cloud replication latency, and consistent access control across environments is deeply complex. Tools like Terraform, Crossplane, and Debezium CDC help but the operational surface area remains vast.
04
AI & Vector Workloads
LLMs require vector databases to store and query high-dimensional embeddings for semantic search and RAG (Retrieval Augmented Generation). Dedicated solutions like Pinecone, Weaviate, and Qdrant compete with vector extensions in PostgreSQL (pgvector) and MongoDB Atlas. Integrating AI inference directly into the database query layer is the emerging frontier.
05
Operational Complexity & Observability
Modern stacks run dozens of database types simultaneously. Understanding slow queries, lock contention, connection pool exhaustion, and replication lag across heterogeneous systems demands sophisticated observability. Platforms like Datadog, pganalyze, and VictoriaMetrics provide the monitoring layer; but database observability remains an unsolved problem for many teams.
